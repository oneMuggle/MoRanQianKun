## Context

角色锚点功能后端逻辑已完成（数据结构定义、CRUD 操作、AI 提取服务、生图装配），但 UI 层仅部分实现。桌面端 `ImageManagerModal.tsx` 有基础锚点区域展示，移动端尚未接入。需要补全两端的完整管理 UI，包括列表、详情编辑、提取操作和删除功能。

### 当前已有

- `角色锚点结构` 类型定义（models/system.ts）
- 角色锚点 CRUD 操作（hooks/useGame.ts）
- AI 提取角色锚点服务（hooks/useGame.ts → extractCharacterAnchor）
- 桌面端部分 UI：
  - 锚点列表展示
  - 锚点提取按钮
  - 锚点保存/删除按钮
  - 基础开关

### 当前缺失

- 移动端角色锚点管理 UI
- 桌面端锚点详情编辑（编辑正负面提示词、结构化标签、各项开关）
- 锚点管理入口的更明显展示
- 锚点列表的分页/筛选

## Goals / Non-Goals

**Goals:**
- 桌面端和移动端均能完整管理角色锚点
- 支持锚点的增删改查全生命周期
- 支持从 NPC 资料一键提取锚点
- 支持锚点各项开关配置（启用、默认附加、场景自动注入）

**Non-Goals:**
- 修改角色锚点的后端存储逻辑
- 修改角色锚点提取提示词
- 修改图片生成装配逻辑
- 修改词组转化器预设管理

## Decisions

### 1. UI 入口选择：复用图片管理而非新建独立面板

**决策**: 在现有 `ImageManagerModal`（桌面端）和 `MobileImageManagerModal`（移动端）中扩展锚点管理 Tab。

**理由**:
- 锚点与图片生成强相关，用户在管理生图时自然需要管理锚点
- 避免新增独立面板增加导航复杂度
- 现有桌面端已有部分锚点 UI，只需扩展

**备选**: 新建独立 `CharacterAnchorModal` - 优点是职责更清晰，缺点是增加入口成本

### 2. 锚点列表展示形式：卡片式列表

**决策**: 采用卡片列表形式，每张卡片显示锚点名称、关联 NPC、启用状态、场景联动标签、正面提示词预览。

**理由**:
- 卡片形式适���展示多字段信息
- 可直观显示启用状态和特殊标签
- 符合现有 UI 风格（预设管理、画师串管理等）

### 3. 锚点详情编辑：内联编辑 vs 弹窗编辑

**决策**: 详情在列表下方展开编辑区（类似现有词组转化器预设编辑方式）。

**理由**:
- 保持操作连续性，无需切换弹窗
- 减少用户认知负担
- 符合现有组件的交互模式

### 4. 移动端适配：独立移动端组件

**决策**: 创建 `MobileImageManagerModal.tsx` 中的角色锚点管理区，遵循移动端交互模式。

**理由**:
- 移动端屏幕空间有限，需要不同布局策略
- 遵循现有模块的桌面/移动端分离模式

## Risks / Trade-offs

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 移动端空间有限，展示内容受限 | 信息密度不足 | 采用可折叠详情，只显示关键摘要 |
| 结构化标签编辑复杂 | 用户体验差 | 提供批量编辑入口，默认保持 AI 提取结果 |
| 锚点与 NPC 绑定后无法迁移 | 灵活性不足 | UI 提示「每个 NPC 只能有一个锚点」 |

## Impact Assessment

### 功能影响
- 用户可完整管理角色锚点，无需直接操作 IndexedDB
- 提升角色生图一致性的可配置性

### 兼容性
- 不影响已有存档
- 不影响已有生图链路
- 仅扩展 UI，不改变数据模型

### 性能
- 锚点列表无分页时直接渲染全部（预计 <100 个锚点）
- 如需分页，添加虚拟滚动优化
