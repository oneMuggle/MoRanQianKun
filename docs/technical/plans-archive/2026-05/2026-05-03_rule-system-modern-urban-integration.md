# 规则系统融入现代都市纪元计划

## 背景与目标

**来源文件：**
- `references/规则模拟器V1.4.json` — SillyTavern 角色卡，包含规则引擎元系统（世界规则、区域规则、个人规则、变量管理、叙事控制边界）
- `references/规则index.html` — 规则系统可视化管理 UI（支持7种世界类型，含现代都市默认场景）

**现状分析：**
- 项目已有完整的 `contemporary_urban`（现代都市）子纪元系统，包含 6 种职业预设（大厂员工、网约车司机、外卖骑手、理发师、装修师傅、便利店老板）
- 每种职业已配有对应的天赋、背景、气运自动填充
- 但缺少**世界规则层**的规则预设——即现代都市特有的环境规则（如交通规则、互联网文化、职场等级、社交媒体影响等）
- 现有节日系统（`data/world.ts`）为传统节日，缺少现代都市节日

**目标：**
将规则模拟器的规则框架融入到现代都市纪元中，使玩家选择现代都市职业后，自动获得与该职业匹配的世界规则、区域规则和个人规则，增强沉浸感和叙事一致性。

## 涉及的文件与模块

| 文件 | 作用 | 变更类型 |
|------|------|----------|
| `data/presets.ts` | 天赋、背景、气运定义 | 新增现代规则相关天赋/气运 |
| `data/world.ts` | 世界数据（节日等） | 新增现代节日 |
| `data/subEraDefaultPresets.ts` | 子纪元默认预设 | 扩展现代预设的规则配置 |
| `types.ts` | 类型定义 | 可能需要扩展预设结构以支持规则引用 |
| `components/features/NewGame/NewGameWizardContent.tsx` | 新建游戏 UI | 规则选择/展示 UI |
| `components/features/NewGame/useNewGameWizardState.ts` | 新建游戏状态管理 | 规则状态管理 |

## 技术方案

### 核心设计思路

规则模拟器定义了三类规则：
1. **世界规则（World Rules）** — 全局生效，成为常识（如"交通规则"、"互联网普及"）
2. **区域规则（Area Rules）** — 特定区域内生效（如"写字楼打卡规则"、"外卖配送范围"）
3. **个人规则（Personal Rules）** — 针对特定角色（如"社畜的加班体质"）

现代都市纪元的融入策略：**按职业绑定规则预设**，选择职业时自动加载匹配的规则集。

## 实施步骤

### Phase 1: 扩展类型定义 (Low)
- [ ] 在 `types.ts` 中扩展 `子纪元默认预设结构`，新增可选字段：
  - `世界规则名称列表?: string[]`
  - `区域规则名称列表?: string[]`
  - `个人规则名称列表?: string[]`

### Phase 2: 定义现代都市规则数据 (Medium)
- [ ] 在 `data/presets.ts` 或新建 `data/rules/modernUrbanRules.ts` 中定义规则数据结构：
  - `规则条目` 接口（名称、效果描述、分类、适用职业、SFW/NSFW标记）
  - 世界规则数组（5条通用规则）
  - 区域规则数组（6个职业 x 4条 = 24条）
  - 个人规则数组（6个职业 x 4条 = 24条）
  - NSFW规则数组（4条）

### Phase 3: 扩展天赋与气运 (Medium)
- [ ] 在 `data/presets.ts` 中为每个现代都市职业添加配套天赋（如果当前不足）：
  - 确保每个职业的 2-3 个天赋与对应规则产生叙事联动
  - 例：网约车司机的"平台算法理解"天赋 ↔ "平台抽成规则"区域规则
- [ ] 在 `data/qiyun/index.ts` 中为每个现代都市职业添加配套气运：
  - 确保气运名称与规则系统产生呼应
  - 例：外卖骑手的"五星好评"气运 ↔ "乘客评分系统"区域规则

### Phase 4: 更新子纪元默认预设 (Low)
- [ ] 在 `data/subEraDefaultPresets.ts` 中为每个 `contemporary_urban` 预设添加规则引用：
  - `世界规则名称列表: ['现代法律框架', '互联网基础设施', '货币经济体系', '社交网络影响', '信息时代节奏']`
  - 按职业添加对应的区域规则和个人规则名称列表

### Phase 5: 新增现代都市节日 (Low)
- [ ] 在 `data/world.ts` 的节日列表中新增现代节日：
  - 双十一购物节、情人节（商业化）、毕业季、金三银四招聘季、国庆黄金周、春运

### Phase 6: UI 展示层 (Medium)
- [ ] 在新建游戏向导中，选择职业后显示该职业关联的规则预览
- [ ] 在游戏内世界书/规则书中展示当前生效的规则列表
- [ ] 确保规则信息在移动端和桌面端都有合适的展示方式

### Phase 7: 叙事系统集成 (High)
- [ ] 在 `prompts/` 目录下新增或更新规则相关 prompt，使 AI 叙事时引用当前生效的规则
- [ ] 在 `hooks/useGame/systemPromptBuilder.ts` 中注入规则上下文
- [ ] 确保规则变化能够影响叙事走向（如违反交通规则导致交警罚款事件）

## 风险评估与依赖

### 风险
- **LOW**: 类型定义扩展 — 纯代码变更，无外部依赖
- **MEDIUM**: 规则数据设计 — 需要平衡 SFW 和 NSFW 内容，确保规则描述有足够的叙事引导性
- **MEDIUM**: UI 展示 — 规则信息量较大，需要合理的折叠/展开设计
- **HIGH**: 叙事系统集成 — 需要修改 prompt 模板，可能影响现有故事生成的稳定性

### 依赖
- 无外部依赖，全部为项目内部代码
- 需要与现有 `contemporary_urban` 预设保持向后兼容

## 估计复杂度

- **总复杂度：MEDIUM-HIGH**
- Phase 1-2: 1-2 小时（数据定义）
- Phase 3-4: 1-2 小时（预设扩展）
- Phase 5: 0.5 小时（节日添加）
- Phase 6: 2-3 小时（UI 开发）
- Phase 7: 3-4 小时（叙事集成）
- **总计：7.5-11.5 小时**
