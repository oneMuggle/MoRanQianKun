# 云同步实现与配置分析

## 一、架构概览

该项目使用 **GitHub 私有仓库 + Release 附件** 作为云同步后端，存储位置是 GitHub Release 的分卷上传附件。

技术栈：
- **压缩库**：fflate（ZIP 打包/解包）
- **存储后端**：GitHub 私有仓库 + Release 附件
- **认证**：GitHub OAuth（或 Personal Access Token）
- **代理**：Cloudflare Pages Functions（或等效服务端）

---

## 三、同步原理

### 3.1 数据打包

使用 `fflate` 库将以下内容打包为 ZIP：

```
WuXia_Save_Data.zip/
├── manifest.json           # 清单元数据
├── saves/
│   ├── index.json        # 存档索引
│   └── manual_*.json    # 手动存档
│   └── auto_*.json      # 自动存档
├── settings/
│   ├── index.json      # 设置索引
│   └── *.json          # 各类设置
└── assets/bin/
    ├── index.json      # 图片资源索引
    └── *.jpg/png     # 图片二进制
```

### 3.2 分卷上传

大文件自动切分为 **8MB 分卷**，解决 GitHub 单个附件 2GB 限制：

```typescript
const 云同步分卷大小 = 8 * 1024 * 1024; // 8MB
```

### 3.3 存储位置

- **仓库**：自动在用户 GitHub 账号下创建私有仓库（默认名 `wuxia-cloud-save`）
- **Release 标签**：`wuxia-cloud-sync`
- **附件**：分卷 + 清单文件（`.parts.json`）

### 3.4 核心 API

```typescript
// services/githubSync.ts
await uploadToCloud(token, onProgress)    // 上传
await downloadFromCloud(token, onProgress) // 下载恢复
await fetchSyncMetaData(token)           // 查询远端状态
await restoreSyncData(zipBytes)           // 恢复数据
```

## 七（一）、部署到 GitHub Pages

### 7.1 架构限制

**GitHub Pages 是纯静态托管，不支持服务端 API**。

当前云同步依赖的后端 API：
- `/api/auth/github` - OAuth code 换 token
- `/api/github/release-upload` - Release 附件上传代理
- `/api/github/release-download` - Release 附件下载代理

这些 API **无法在 GitHub Pages 上运行**。

### 7.2 两种解决方案

#### 方案 A：使用 Cloudflare Workers（免费）

将后端 API 部署到 Cloudflare Workers，前端部署到 GitHub Pages：

```bash
# 1. 部署后端 API 到 Cloudflare Workers
wrangler deploy functions/api --env production

# 2. 前端配置指向 Workers
VITE_API_BASE=https://your-worker.xxx.workers.dev
```

#### 方案 B：使用 Vercel（推荐）

将前端和后端 API 一起部署到 Vercel：

```bash
# vercel.json 配置
{
  "rewrites": [
    { "source": "/api/(.*)", "destination": "/api/$1" }
  ]
}
```

### 7.3 GitHub Pages + 外部 API 配置

#### 前端配置（.env）

```bash
# .env.production
VITE_GITHUB_CLIENT_ID=your_client_id_here

# 指向后端 API（Cloudflare Workers / Vercel / 自建服务器）
VITE_API_BASE=https://your-api-domain.com
```

#### OAuth Callback URL

在 GitHub OAuth App 设置中：
- **Authorization callback URL**: `https://your-username.github.io/repo-name`

### 第一阶段：GitHub OAuth App 配置

#### 步骤 1.1：创建 GitHub OAuth App

1. 打开 https://github.com/settings/developers
2. 点击 **New OAuth App**
3. 填写表单：

| 字段 | 填写示例 | 备注 |
|------|----------|------|
| Application name | `墨色江湖-云同步` | 可自定义 |
| Homepage URL | `https://your-name.pages.dev` | 暂时先用这个，后续可改 |
| Application description | `武侠游戏的云存档同步功能` | 可选 |
| Authorization callback URL | `https://your-name.pages.dev` | **必须与部署地址一致** |

4. 点击 **Register application**
5. 记录 **Client ID**（页面顶部）
6. 在 **Client secrets** 区域点击 **Generate a new client secret**
7. 记录生成的 **Client Secret**（只会显示一次！）

> ⚠️ **重要**：保存好 Client Secret，后续需要用到。

### 第二阶段：Cloudflare 账号准备

2. 用邮箱注册账号

4. 点击 **创建自定义令牌**
5. 配置：

|------|------|
| 名称 | `wuxia-deploy` |
| 权限 | 选择 **账户** → **Cloudflare Pages** → **编辑** |

| 有效期 | 永久 或 自定义 |

6. 点击 **创建令牌**
7. 复制显示的 **API Token**（只会显示一次！）

> ⚠️ **重要**：保存好 API Token，后续 GitHub Actions 需要用到。

#### 步骤 2.3：获取 Cloudflare Account ID

1. 登录 Cloudflare Dashboard
2. 点击右侧头像旁边的 **账户** 下拉菜单

### 第五阶段：创建 GitHub Actions 部署脚本

#### 步骤 5.1：创建部署脚本

在项目根目录创建 `.github/workflows/deploy.yml`：

```yaml
name: Deploy to Cloudflare Pages

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      deployments: write

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Build
        run: npm run build

      - name: Deploy to Cloudflare Pages
        uses: cloudflare/pages-action@v1
        with:
          apiToken: ${{ secrets.CLOUDFLARE_API_TOKEN }}
          accountId: ${{ secrets.CLOUDFLARE_ACCOUNT_ID }}
          projectName: wuxia-game
          directory: dist
          # 可选：同时部署 API 函数
          # wrangler.toml 配置了 apiDirectory 时使用
```

#### 步骤 5.2：创建 wrangler.toml（可选）

在项目根目录创建 `wrangler.toml`：

```toml
name = "wuxia-game"
compatibility_date = "2024-01-01"

# Pages 配置
[site]
bucket = "./dist"

# API 函数
pages_build = { dir = "dist", runtime = "off" }

# 如果要部署 API 函数，创建 _worker.js 或使用 functions 目录
# Cloudflare Pages Functions 会自动识别 functions/ 目录下的 .ts 文件
```

### 常见问题排查

#### Q1：部署失败，提示权限不足

**解决**：检查 Cloudflare API Token 权限，确保包含 **Cloudflare Pages** 的 **编辑** 权限。

#### Q2：云同步功能无法使用

**排查步骤**：

1. 打开浏览器开发者工具 → Console
2. 点击"云存档登录"
3. 查看是否有错误信息

常见错误：
- `未配置 VITE_GITHUB_CLIENT_ID` → 确保环境变量配置正确
- `CORS 错误` → 检查 Cloudflare 环境变量是否正确配置
- `401 Unauthorized` → 检查 GitHub OAuth App 的 Client Secret

#### Q3：OAuth 回调失败

**解决**：
1. 确认 GitHub OAuth App 的 **Authorization callback URL** 与实际访问地址一致
2. 检查 Cloudflare 环境变量中 `GITHUB_CLIENT_ID` 是否正确

#### Q4：Token 过期或失效

**解决**：重新生成 GitHub OAuth App 的 Client Secret，并在 Cloudflare 中更新。

如无 OAuth App，可使用 **Personal Access Token** 直接操作：

1. 获取 Token：GitHub → Settings → Developer settings → Personal access tokens
2. 首次上传时选择"直接使用 Token"模式
3. 输入 Token 后即可使用

## 十二、OAuth 配置与用户登录授权的区别

### 12.1 两个不同的概念

| 概念 | 说明 | 谁做 |
|------|------|-----|
| **OAuth 配置**（`VITE_GITHUB_CLIENT_ID`） | 开发者在 GitHub 注册的 OAuth App 身份凭证 | 开发者（一次性） |
| **用户登录授权** | 每次用户点击登录时，GitHub 询问是否允许访问 | 用户（每次） |

### 12.2 流程对比

```
OAuth App 配置（开发者做一次）
┌─────────────────────────────────────┐
│  开发者去 GitHub 创建 OAuth App          │
│  → 获得 Client ID + Client Secret     │
└─────────────────────────────────────┘
                    ▼
用户登录授权（每个用户每次登录）
┌─────────────────────────────────────┐
│  1. 用户点击"云存档登录"             │
│  2. 跳转到 GitHub 授权页面         │
│  3. 用户点击"Authorize xxx"       │
│  4. GitHub 返回 code              │
│  5. 用 code 换 access_token      │
│  6. Token 存入 localStorage       │
└─────────────────────────────────────┘
```

### 12.3 两者分工

| 配置 | 用途 |
|------|------|
| `VITE_GITHUB_CLIENT_ID` | **登录授权** - 点击"云存档登录"后跳转到 GitHub OAuth 页面 |
| 仓库名 | **数据存储** - 首次上传时输入，指定私有仓库名 |

## 十三、当前设计缺陷

### 13.1 问题

当前代码在 `hooks/useGitHubOAuth.ts:60-67`：

```typescript
const login = () => {
    const clientId = (import.meta as any).env.VITE_GITHUB_CLIENT_ID;
    if (!clientId) {
        alert('未配置 VITE_GITHUB_CLIENT_ID 环境变量');
        return;  // ← 直接退出，没有替代方案
    }
    window.location.href = `https://github.com/login/oauth/authorize?client_id=${clientId}&scope=repo`;
};
```

当没有配置 `VITE_GITHUB_CLIENT_ID` 时，代码直接退出，没有提供替代登录方式。

### 13.2 死胡同

```
如果无 OAuth 配置
    → alert 退出
    → 无法登录
```

应该改为：

```
如果无 OAuth 配置
    → 让用户手动输入 Token（PAT）
```

### 13.3 临时解决方案

在浏览器控制台手动设置 Token：

```javascript
// 在浏览器 console 执行
localStorage.setItem('github_sync_token', 'YOUR_GITHUB_TOKEN')
location.reload()
```

获取 Token：
1. GitHub → Settings → Developer settings
2. Personal access tokens → Tokens (classic)
3. Generate new token（勾选 `repo` 权限）

## 十四、多用户隔离机制

每个用户的云同步是**独立的**：

1. **每个用户独立仓库**：首次使用时会弹出 `window.prompt` 让用户输入仓库名
2. **绑定到用户账号**：仓库创建在当前 GitHub 账号下
3. **本地存储**：仓库名 + Token 存储在 `localStorage` 中

```typescript
// services/githubSync.ts:313-317
const 获取仓库配置 = async (token: string): Promise<RepoConfig> => {
    const user = await 获取当前用户(token);  // 获取当前 GitHub 用户
    const repo = 获取或询问仓库名();            // 用户输入仓库名
    return { owner: user.login, repo };    // owner=用户名, repo=仓库名
};
```

不同用户的数据互不影响，因为存储在各自 GitHub 账号下的私有仓库中。
