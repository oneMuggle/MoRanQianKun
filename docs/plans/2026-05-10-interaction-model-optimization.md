# 交互模式优化方案

> 日期：2026-05-10
> 状态：规划中
> 触发：评估除现有交互模式外的更优展示方案

---

## 1. 当前交互模式概述

### 视图架构

应用有三个顶级视图，由 `state.view` 控制：

| 视图 | 组件 | 用途 |
|------|------|------|
| `home` | `LandingPage` | 入口：开始游戏、加载存档、管理图片/世界书 |
| `new_game` | `NewGameWizard` / `MobileNewGameWizard` | 角色创建和世界生成 |
| `game` | `GameView` | 主游戏循环 |

### 桌面端布局（三栏"仪表板+聊天"）

```
+------------------------------------------------------------------+
|  TopBar (天气 | 时间/地点 | 节日 | 旅程天数)                       |
+------------------+-----------------------+------------------------+
|  LeftPanel       |  Chat Area (中心)     |  RightPanel            |
|  - 角色名/头像   |  - 可滚动历史         |  - 14+ 菜单按钮         |
|  - 状态条        |    (TurnItem 卡片)    |    (战斗、装备、背包等) |
|  - 身体部位      |  - 输入框             |  - 存档/设置            |
|  - 装备          |  - 快捷操作按钮       |                        |
|  - BUFFs         |  - 管线进度指示       |                        |
+------------------+-----------------------+------------------------+
|  Ticker bar (滚动: "世界事件")                                    |
+------------------------------------------------------------------+
```

### 移动端布局（单栏 + 底部导航）

```
+----------------------------------------+
|  TopBar (折叠: 天气/环境 + 时间)         |
+----------------------------------------+
|                                        |
|         Chat Area (全宽)               |
|         + 输入框 + 快捷按钮             |
|                                        |
+----------------------------------------+
|  快捷菜单 (6图标 + 更多)                |
|  [角色][战斗][装备][背包][社交][更多]     |
+----------------------------------------+
|  Ticker bar (滚动)                      |
+----------------------------------------+
```

### 核心交互模式

主要参与模型是**聊天式回合交互**：
1. 玩家输入自由文本或点击快捷操作按钮
2. AI 处理请求（流式响应，含"思考"阶段）
3. 结构化响应渲染为 `TurnItem` 卡片
4. 后台管线运行：文本润色、变量生成、世界演化、规划
5. 所有游戏状态通过侧面板触发的模态框访问

### 现有模式优势

- 经 AI 驱动互动叙事验证的模式（类似 AI 聊天应用）
- 侧面板提供一目了然的状态，不遮挡叙事
- 模态框保持聊天区域整洁
- 代码库中领域分离良好

### 现有模式劣势

- **以聊天为中心限制了空间想象**：世界/地图被简化为文本，玩家无法视觉探索
- **模态框打断流程**：打开任何功能都会从叙事中转移注意力
- **信息密度权衡**：左右面板始终可见但经常与当前时刻无关
- **移动端面板丢失**：侧面板在移动端消失，隐藏关键角色状态
- **无叙事结构视图**：聊天是线性滚动，无法查看故事弧、章节或时间线
- **地图数据未充分利用**：地图节点作为数据存在但以简单列表显示
- **被动参与**：玩家主要是阅读和输入，没有视觉交互、探索或空间决策

## 2. 替代展示方案评估

### 方案 A：全屏叙事视图（视觉小说风格）

**描述**：将三栏布局替换为全屏叙事画布。侧面板变为滑出式覆盖层。聊天区成为主导视觉元素，配合角色立绘和场景背景。

**工作方式**：
- 全屏故事文本，类似视觉小说（自上而下或居中）
- 对话时角色立绘出现在侧边
- 场景图片填充视口作为背景
- 左面板（角色状态）按需从左侧滑入
- 右面板（菜单）从右侧滑入或显示为径向菜单
- 顶部信息变为微妙的状态丝带
- 输入区保留在底部

**武侠主题契合度**：**非常高**
- 武侠小说有强烈的视觉意象（场景、角色、武打），适合全屏展示
- 视觉小说格式与中国互动叙事文化高度契合
- 场景背景+角色立绘增强沉浸感，不改变核心文本玩法

**实现难度**：**中等**
- `GameView.tsx` 已支持背景图片和隐藏聊天内容
- LeftPanel 和 RightPanel 已是独立组件，可改为抽屉式
- 需要新的 `NarrativeView` 组件用于全屏故事展示
- 模态系统已存在，可处理滑出面板
- 预估：3-5 个新建/修改组件，约 2-3 周

**迁移风险**：**低**
- 可作为"沉浸模式"切换开关引入，与现有布局共存
- 现有模态框和聊天渲染保持不变
- 可在 A/B 测试后决定是否设为默认

**优势**：
- 大幅提升叙事沉浸感
- 更好利用已有的场景图片生成系统
- 为叙事文本提供更多屏幕空间
- 减少阅读时的视觉干扰
- 移动端受益最大（已全宽，获得面板覆盖层）
- 与武侠视觉小说美学一致

**劣势**：
- 状态可见性降低，需要玩家主动查看
- 背景图片可能影响文字可读性（可通过遮罩渐变缓解）

---

### 方案 B：地图驱动探索视图

**描述**：以可探索地图替代聊天优先模型。地图显示位置节点和连接路径。玩家点击/拖拽探索地点，聊天变为次要面板。

**工作方式**：
- 交互地图渲染 `世界.地图` 节点为可定位的点/区域
- 玩家点击节点移动（点击旅行）
- 进入地点触发场景（叙事文本出现在侧面板或覆盖层）
- 地图节点可有视觉状态（已探索/未探索、敌对/安全）
- 建筑作为子节点
- 聊天面板从底部或侧边滑入
- 世界演化更新地图（新节点出现、连接变化）

**武侠主题契合度**：**高**
- 江湖本质上是穿越武侠世界的旅行体验
- 武侠故事与地点深度绑定（寺庙、客栈、山川、城市）
- 地图探索匹配现有旅行/地图功能，但使其成为核心

**实现难度**：**高**
- `MapModal.tsx` 已渲染地图数据但以列表形式，非空间视图
- 需要 Canvas/SVG 地图渲染组件
- 地图节点位置数据需添加到 `地图结构` 模型
- 旅行逻辑需与地图交互集成
- 预估：新地图渲染引擎，5-8 个组件，约 4-6 周

**迁移风险**：**中等**
- 难以与现有布局作为切换共存（结构差异太大）
- 需要作为独立视图模式 (`view === 'explore'`)
- 需要地图数据已填充（当前依赖 AI 生成）

**优势**：
- 将游戏从"阅读和输入"转变为"探索和互动"
- 使世界数据在视觉上产生意义
- 旅行决策变得空间化和直观
- 开启新玩法可能性（隐藏地点、路径选择、区域事件）
- 与标准 AI 聊天应用形成强差异化

**劣势**：
- 需要大量新数据（节点位置、连接、视觉表现）
- AI 生成的地图可能稀疏或不一致
- 实现复杂（寻路、缩放、平移）
- 地图可能喧宾夺主

### 方案 C：时间线/故事板视图

**描述**：添加时间线视图，将线性聊天记录转化为结构化故事时间线，包含章节、故事弧和关键事件。作为聊天视图的补充而非替代。

**工作方式**：
- 时间线视图将故事历史渲染为水平或垂直时间线
- 每个 `TurnItem` 成为时间线上的节点
- 节点按地点、时间段或故事弧分组（自动从上下文检测）
- 点击节点展开完整叙事文本
- 玩家可跳转到时间线任意点重读或分支
- 关键事件（战斗、会面、发现）视觉上突出显示
- 故事小地图显示当前位置相对于完整故事弧

**武侠主题契合度**：**高**
- 武侠故事是章节式的，有清晰的弧线（结识英雄、击败反派、发现秘密）
- 时间线强化"江湖录"的记录感
- 帮助玩家跨多次会话跟踪长期情节线索

**实现难度**：**中等**
- 数据已存在于 `state.历史记录`
- 需要自动将回合聚类为故事弧（可用 AI 或启发式）
- 时间线渲染组件（水平可滚动或垂直）
- 与现有记忆系统集成用于长期故事跟踪
- 预估：2-3 个新组件，约 2-3 周

**迁移风险**：**低**
- 纯新增功能，不修改现有聊天视图
- 可通过右面板或故事模态中的"故事时间线"按钮访问
- 如果无法生成时间线，回退到当前聊天视图

**优势**：
- 解决长会话中"迷失在聊天中"的问题
- 提供线性滚动无法实现的故事总览
- 支持特定故事弧的重玩/重读
- 与记忆系统完美互补
- 低风险，高感知价值

**劣势**：
- 自动聚类可能产生不完美结果
- 增加故事数据模型的复杂性
- 对新/短会话的直接价值有限

---

### 方案 D：卡片式探索 UI

**描述**：用卡片系统替代模态框，游戏信息（角色、物品、NPC、地点）以可交互卡片呈现在可滚动或网格布局中。

**工作方式**：
- 角色信息变为"角色卡"，可翻转（状态/装备）
- 物品是可点击使用/装备的卡片
- NPC 是带有好感度/战斗数据的肖像卡
- 地点是带有描述和动作按钮的卡片
- 卡片可排列成"手牌"（类似卡牌游戏）用于快速决策
- AI 的操作按钮变为可玩卡片

**武侠主题契合度**：**中等**
- 卡牌隐喻契合武侠"武功秘籍"美学
- 对叙事密集内容不太自然
- 更适合物品/管理而非故事阅读

**实现难度**：**中等**
- 将替换或重新设计现有模态内容
- 需要卡片动画/交互库
- 预估：重构 8-10 个模态组件，约 3-4 周

**迁移风险**：**中等**
- 改变所有功能模块的交互模式
- 可增量引入（从物品/角色卡开始）

**优势**：
- 比静态模态框更具交互吸引力
- 卡牌决策契合游戏感
- 移动端触控友好
- 物品管理变得触感化

**劣势**：
- 不改善核心叙事体验
- 卡牌隐喻对文字 RPG 可能显得过于游戏化
- 开发投入相对于 UX 提升较小

---

### 方案 E：仪表盘总览模式

**描述**：添加总览模式，在单个仪表盘视图中展示所有游戏状态信息（类似游戏暂停菜单）。作为补充视图，非替代。

**工作方式**：
- 仪表盘展示：角色状态、当前位置、附近 NPC、活跃任务、物品摘要、最近故事事件、世界状态
- 组织为信息小部件网格
- 每个小部件可点击打开完整功能模态框
- 通过"暂停"或"总览"按钮访问
- 特别适合移动端（侧面板隐藏时）

**武侠主题契合度**：**中等**
- 匹配右面板已有的"天机"概念
- 提供"江湖总览"感
- 不如叙事导向的视图沉浸

**实现难度**：**低**
- 将现有数据组合为新布局
- 无新数据需求
- 预估：1-2 个新组件，约 1-2 周

**迁移风险**：**极低**
- 纯新增，零现有视图变更
- 可独立测试

**优势**：
- 直接解决移动端"隐藏面板"问题
- 快速状态检查，无需打开多个模态框
- 开发成本低
- 作为更激进 UI 变更的良好入口

**劣势**：
- 不根本改变交互模式
- 设计不当可能导致信息过载
- 更偏实用性而非沉浸感

## 3. 推荐方案：分阶段渐进式改进

### 实施路线

| 阶段 | 方案 | 周期 | 价值 | 风险 | 独立性 |
|------|------|------|------|------|--------|
| **Phase 1** | 仪表盘总览模式 (E) | 1-2周 | 快速交付，解决移动端信息隐藏 | 极低 | 完全独立 |
| **Phase 2** | 全屏叙事视图 (A) | 2-3周 | 最大UX提升，复用现有场景图片系统 | 低 | 可作为切换共存 |
| **Phase 3** | 时间线故事板 (C) | 2-3周 | 长期叙事结构，与记忆系统互补 | 低 | 纯新增 |

### 为什么是这个顺序

1. **风险管理**：每个阶段都可独立交付和测试
2. **复合价值**：仪表盘为叙事视图设计提供参考；时间线建立在叙事结构之上
3. **无回退**：所有阶段都是新增而非替换，保留当前可用模型
4. **增量验证**：每个阶段可在提交下一阶段前评估

### 暂不推进的方案

**地图驱动探索 (B)** 是最具变革性但也风险最高的方案。应在以下条件下考虑作为未来大版本：
- 地图数据可靠填充（参见 `2026-05-10-map-feature-optimization.md` 计划）
- 核心叙事 UX 改进（Phase 1-3）已验证
- 空间游戏机制已设计（不仅是 UI）

**卡片式 UI (D)** 与核心文字 RPG 体验的对齐度最低，除非特定游戏系统（如卡牌战斗）需要，否则应降低优先级。

## 4. 实现复杂度汇总

| 方案 | 复杂度 | 周期 | 风险 | 独立性 |
|------|--------|------|------|--------|
| 仪表盘总览 (E) | 低 | 1-2周 | 极低 | 完全独立 |
| 全屏叙事视图 (A) | 中 | 2-3周 | 低 | 可与现有共存 |
| 时间线故事板 (C) | 中 | 2-3周 | 低 | 纯新增 |
| 卡片式 UI (D) | 中 | 3-4周 | 中 | 替换模态框 |
| 地图驱动探索 (B) | 高 | 4-6周 | 中 | 需要新数据 |
