# 写真系统优化计划

**日期**: 2026-05-18
**状态**: 待审批

## 背景与目标

当前写真系统存在以下问题需要优化：

1. **项目识别问题**: 虽然数据结构有 `id` 字段，但 LLM 交互中项目匹配可能依赖名称，导致名称变化后产生重复项目
2. **阶段固定问题**: 拍摄阶段固定为预设序列（化妆造型→第一组拍摄→换装→第二组拍摄→休息→收尾），不能随剧情动态变化
3. **摄影师名称显示问题**: 当摄影师是玩家角色时，UI 显示 "player" 而非主角实际姓名
4. **内容单薄**: 项目内容较简单，缺少摄影集、关系进展变化等交互性内容

## 问题分析

### 问题 1: 项目 ID vs 名称匹配

**根因**: 在 `photographyNSFWIntegration.ts` 中，解析 LLM 返回的状态更新时，项目匹配逻辑可能同时使用 `项目ID` 和 `项目名称` 进行匹配。当 LLM 返回的 `项目名称` 发生变化但 `项目ID` 没有携带或格式不一致时，系统无法匹配到现有项目，于是创建了新项目。

**解决方案**:
- 强化 `id` 作为唯一匹配依据，取消按名称匹配
- 在 prompt 中更明确地强调 `项目ID` 的传递规则
- 对于 LLM 没有提供 `项目ID` 的情况，增加基于 `模特Id + 摄影师Id + 活跃状态` 的 fallback 匹配

### 问题 2: 阶段动态化

**根因**: 当前阶段序列在 `photographyNSFWEngine.ts` 创建项目时硬编码，回合推进时按固定回合数规则切换。

**解决方案**:
- 将固定阶段序列改为动态阶段系统
- 阶段由 LLM 在剧情中驱动，系统只负责记录和验证
- 保留基础的回合计数作为辅助，但不强制绑定阶段
- 阶段转换增加合法性校验（如不能从"未开始"直接跳到"已完成"）

### 问题 3: 摄影师名称显示

**根因**: `PhotographyDashboard.tsx` 中 `PhotographerCard` 显示 `摄影师.姓名`，而摄影师档案中玩家角色的 `姓名` 字段可能未正确初始化或更新。虽然构建了 `摄影师姓名映射`，但映射只在项目卡片中使用，摄影师名录卡片直接用档案数据。

**解决方案**:
- 在 Dashboard 组件中，统一使用 `摄影师姓名映射` 来处理显示名称
- 确保初始化时将玩家角色姓名写入摄影师档案

### 问题 4: 增加交互内容

**分析**: 当前项目主要记录拍摄过程，缺少以下玩家感兴趣的内容：
- **摄影集**: 拍摄完成后的作品集，可以浏览、发布
- **关系进展**: 模特与摄影师之间的信任度、亲密度变化
- **项目关联**: 同一模特/摄影师的多项目关联展示
- **成就系统**: 拍摄里程碑、特殊事件记录

### 数据结构变更

#### 拍摄项目状态 新增字段

```typescript
interface 拍摄项目状态 {
  // ... 现有字段

  // 新增：动态阶段（替代固定序列）
  当前阶段?: string;           // LLM驱动的当前阶段名，如 "外景拍摄"、"棚内打光"、"补妆"
  阶段历史?: { 阶段: string; 回合: number; 描述: string }[];

  // 新增：关系进展
  关系进展?: {
    信任变化: number;          // 本次拍摄信任度变化
    亲密度变化: number;        // 亲密度变化
    默契度: number;            // 0-100，合作默契程度
    关系里程碑?: string[];     // 如 ["首次合作", "突破舒适区", "建立长期合作"]
  };

  // 新增：摄影集
  摄影集?: {
    作品数量: number;
    精选作品?: { 描述: string; 评分: number; 风格: string }[];
    是否发布: boolean;
    发布日期?: string;
    获得好评?: number;
  };
}
```

#### 写真系统扩展 新增字段

```typescript
interface 写真系统扩展 {
  // ... 现有字段

  // 新增：关系网络
  关系网络?: Record<string, {  // key: "模特Id-摄影师Id"
    信任度: number;
    亲密度: number;
    默契度: number;
    合作次数: number;
    关系等级: string;          // 如 "陌生人"、"合作伙伴"、"固定搭档"、"密友"
    里程碑: string[];
  }>;

  // 新增：摄影集索引
  摄影集索引?: Record<string, {  // key: 模特Id 或 摄影师Id
    作品总数: number;
    发布数量: number;
    代表作?: string[];           // 项目ID列表
  }>;

  // 新增：声望系统
  摄影师声望?: Record<string, {  // key: 摄影师Id
    业内声望: number;            // 0-100，技术口碑
    圈内声誉: number;            // 0-100，行业信誉
    风评: '优秀' | '良好' | '一般' | '争议' | '恶劣';
    标签: string[];              // 如 ["专业可靠"、"擅长私房"、"尺度大胆"、"有偷拍传闻"]
    粉丝数量: number;            // 发布作品后的关注者
    合作意向: number;            // 其他NPC主动约拍的概率加成
  }>;

  模特声望?: Record<string, {    // key: 模特Id
    知名度: number;              // 0-100，公开曝光度
    风评: string;                // 如 "清纯模特"、"大胆新星"、"争议人物"
    标签: string[];              // 如 ["实力派"、"花瓶"、"敢拍敢拼"]
    黑料风险: number;            // 0-100，被泄露/曝光的潜在威胁值
  }>;
}
```

#### 声望对AI叙事的影响

声望系统需要反馈到主游戏的 prompt 中，影响 SFW 和 NSFW 叙事：

| 声望维度 | SFW 影响 | NSFW 影响 |
|----------|----------|-----------|
| 摄影师-业内声望 | NPC主动约拍事件频率、商业合作剧情 | 摄影师接触高端/边缘项目的概率 |
| 摄影师-风评 | 社交圈中的口碑对话、推荐事件 | 模特信任度初始值、越界倾向评估 |
| 模特-知名度 | 街头被认出、粉丝互动事件 | 泄露后传播速度、舆论压力 |
| 模特-黑料风险 | 日常生活中的焦虑/防备剧情 | 泄露事件触发概率、勒索/BDSM联动 |

## 风险评估与依赖

| 风险 | 等级 | 应对 |
|------|------|------|
| LLM 不遵守 ID 传递规则 | 中 | 增加容错匹配 + prompt 强化 |
| 动态阶段导致状态不一致 | 低 | 增加阶段转换校验 |
| 新增字段与现有存档兼容 | 低 | 所有新字段可选，提供默认值 |
| NSFW 维度过多导致 prompt 膨胀 | 高 | 分层注入，仅活跃维度进入 prompt |
| 维度联动逻辑过于复杂 | 中 | 分阶段实现，每阶段独立验证 |
| LLM 无法同时处理多个 NSFW 线索 | 中 | 每回合限定最多激活 2-3 条线索 |

**依赖**: 无外部依赖，纯代码逻辑变更。
