# 写真项目重复记录修复方案

## 背景

写真约拍NSFW系统中，同一个写真项目会因为对话过程中项目名称改变，导致被记录为多个重复项目。

## 根因分析

问题出在 `hooks/useGame/photographyNSFWIntegration.ts` 的 `应用写真系统状态更新` 函数中（约260-337行）。

### 当前匹配逻辑

```
1. 有 ID → 按 ID 精确匹配
2. 无 ID + 有名称 → 按名称精确匹配（仅当无 ID 时）
3. 都匹配不到 → 创建新项目
```

### 问题场景

1. LLM 在对话中修改了项目名称（如 "春日外景写真" → "春日外景写真-追加夜景"）
2. LLM 没有正确输出 `id` 或 `项目ID` 字段，或输出格式不正确
3. 名称已改变，精确匹配失败
4. 走入 `else` 分支，创建了一个新项目
5. 同一拍摄项目被记录为多个

## 修复方案

### 修改文件

`hooks/useGame/photographyNSFWIntegration.ts`

### 增强匹配逻辑

```
1. 有 ID → 按 ID 精确匹配（保持不变）
2. 无 ID + 有名称 → 按名称精确匹配（保持不变）
3. 无 ID + 名称匹配不到 → 按 (模特Id + 摄影师Id) 组合 + 名称相似度匹配（新增）
4. 仍然匹配不到 → 创建新项目（保持不变）
```

### 新增策略：名称相似度回退匹配

当 ID 和名称精确匹配都失败时：
- 检查更新中是否包含 `模特Id` 和 `摄影师Id`
- 在现有项目中查找人员和模特/摄影师都匹配的项目
- 对名称进行相似度判断：旧名称包含新名称，或新名称包含旧名称
- 如果匹配成功，认为是同一项目，使用现有索引
- 记录日志便于调试

### 代码变更

在 `应用写真系统状态更新` 函数的第277行之后（名称精确匹配失败后），添加名称相似度回退匹配逻辑。

## 风险评估

- **复杂度：** 低
- **风险：** 低 — 只增加匹配条件，不改变数据结构或现有逻辑
- **回退方案：** 如果出现问题，移除新增的匹配代码即可恢复
