# 墨染乾坤：万象纪元 - Godot 移植方案

> 移植目标：将现有 Web 文字 RPG 改造为类似《这是我的战争》视角的 Godot 游戏

## 一、可行性分析

### 结论：**可行，但需要大幅缩小首期范围**

当前项目是一个极其庞大的 AI 驱动文字 RPG 系统（32 万行 TypeScript，320+ 工作流文件，180+ 数据模型）。直接全量移植不现实，但可以**分阶段渐进式移植**，核心逻辑和数据模型可以复用，UI 和部分需要完全重写。

### 核心优势（可直接复用的部分）

| 类别 | 内容 | 移植难度 |
|------|------|----------|
| 数据模型 | 角色、NPC、装备、物品、功法、门派等 180+ 类型定义 | **低** - 转为 GDScript 类/Resource |
| 提示词系统 | 90+ 提示词模板（纯文本） | **极低** - 直接转为 JSON/文本资源 |
| 战斗计算逻辑 | 回合制伤害、buff、先攻计算 | **中** - 转写为 GDScript |
| 世界演化逻辑 | 时间推进、事件调度 | **中** - 需转写 |
| 存档格式 | JSON 序列化数据结构 | **低** - Godot 原生支持 JSON |
| 数据驱动内容 | 天赋、背景、气运、修炼体系预设 | **低** - 转为 JSON/Resource |

### 需要完全重写的部分

| 类别 | 内容 | 重写难度 |
|------|------|----------|
| 所有 React UI 组件 | 200+ 组件（Chat、Modal、Panel 等） | **高** - Godot Control 节点重写 |
| Web 持久化 | IndexedDB | **中** - 改用 FileAccess/ConfigFile |
| AI 服务层 | 浏览器 HTTP 流式调用 | **中** - Godot HTTPRequest 替代 |
| Tailwind CSS | 全部样式系统 | **高** - Godot Theme/StyleBox |
| 状态管理 | Zustand + React hooks | **中** - Godot Autoload/Signal |

## 二、《这是我的战争》核心特点分析

### 1. 视角设计

| 特点 | 说明 | 对墨染乾坤的启示 |
|------|------|------------------|
| **平民视角** | 不扮演英雄，而是普通人在极端环境中求生 | 墨染乾坤的「角色」可以代入类似视角，尤其在古代/原始纪元 |
| **2D 横截面视图** | 侧视剖切建筑内部，营造「蚁观」感 | 可用 Godot 2D 实现类似效果，展示角色居所/门派/避难所 |
| **信息受限** | 黑夜/迷雾限制视野，增强紧张感 | 可用于探索场景，配合墨染乾坤的「地图探索」系统 |

### 2. 核心玩法循环

```
┌─────────────┐    分配任务     ┌─────────────┐
│   白天阶段   │ ──────────────→ │   夜晚阶段   │
│  避难所管理   │                 │  外出探索     │
│  休息/制作   │ ←────────────── │  搜刮/战斗   │
│  照顾伤员    │    返回+结算     │  潜行/交易   │
└─────────────┘                 └─────────────┘
```

**移植适配**：墨染乾坤可以保留这个循环，但将「搜刮」替换为「历练/探索」，将「制作」替换为「修炼/炼丹/炼器」，将「避难所」替换为「居所/洞府/门派」。

### 3. 关键机制

| 机制 | 这是我的战争 | 墨染乾坤适配方案 |
|------|-------------|-----------------|
| 资源管理 | 食物、水、药品、材料 | 灵石、丹药、药材、炼器材料 |
| 心理状态 | 心情值影响工作效率 | 心境/心魔影响修炼效率和突破 |
| 道德抉择 | 偷窃/杀戮导致角色崩溃 | 正邪选择影响气运、宗门声望 |
| 角色差异化 | 帕夫列跑得快、布鲁诺善烹饪 | 不同灵根/天赋影响修炼方向 |
| 基地建设 | 床、雨水收集器、工作台 | 洞府灵田、炼丹炉、炼器台、阵法 |
| 时间压力 | 狙击手限制白天外出 | 天劫倒计时/宗门任务期限 |

## 三、建议移植方案（分 5 阶段）

### 阶段一：核心骨架（4-6 周）

**目标**：实现基础 RPG 框架 + 《这是我的战争》风格的 2D 视图

#### 1.1 项目搭建
- [ ] 创建 Godot 4.x 项目（建议 Godot 4.3+）
- [ ] 配置项目结构：`scenes/`, `scripts/`, `assets/`, `resources/`, `data/`
- [ ] 建立 Autoload 全局单例：`GameManager`, `AIManager`, `SaveManager`

#### 1.2 数据模型移植
- [ ] 将核心 TypeScript 接口转为 GDScript 类/Resource：
  - `角色数据结构` → `Character.gd` (继承 Resource)
  - `物品/装备` → `Item.gd`, `Equipment.gd`
  - `世界状态` → `WorldState.gd`
- [ ] 将天赋、背景、修炼体系预设转为 JSON/Resource

#### 1.3 2D 横截面场景（核心视觉）
- [ ] 搭建「居所」场景（类似避难所的侧视剖切视图）
  - 使用 TileMap 构建多层建筑
  - 添加房间功能区域（卧室、炼丹房、练功房、灵田）
- [ ] 实现角色在居所内的 2D 移动和交互
- [ ] 添加日夜循环视觉效果（白天明亮、夜晚昏暗+灯光）

#### 1.4 基础 UI
- [ ] 顶部状态栏（角色属性、资源、时间）
- [ ] 底部操作面板（分配任务、制作、休息）
- [ ] 侧边栏（角色列表、背包）
- [ ] 中央对话/叙事面板（AI 生成的文字描述）

### 阶段二：AI 叙事层（3-4 周）

**目标**：移植提示词系统，实现「玩家输入 → AI 生成 → 状态更新」循环

#### 2.1 AI 服务接入
- [ ] 使用 Godot HTTPRequest 实现多供应商 API 调用
- [ ] 支持流式响应（SSE）
- [ ] 配置 API 设置界面（复用现有 `apiConfig.ts` 逻辑）

#### 2.2 提示词系统移植
- [ ] 将 90+ 提示词文件转为 JSON 资源
- [ ] 实现提示词组装逻辑（systemPromptBuilder.ts → GDScript）
- [ ] 实现 AI 响应解析（JSON 修复 + 结构化提取）

#### 2.3 核心叙事循环
- [ ] 玩家文字输入 → 调用 AI → 解析响应 → 更新游戏状态
- [ ] AI 响应与 2D 场景联动（AI 描述的事件触发场景动画）

### 阶段三：玩法系统（6-8 周）

**目标**：实现《这是我的战争》风格的核心玩法

#### 3.1 昼夜循环
- [ ] 白天阶段：居所内分配任务（修炼、炼丹、休息、照顾伤员）
- [ ] 夜晚阶段：外出探索（2D 横版关卡，潜行+搜刮+遭遇战）
- [ ] 时间管理系统（每回合消耗时间，触发世界事件）

#### 3.2 资源与制作
- [ ] 资源系统（灵石、药材、矿石、符纸等）
- [ ] 制作系统（炼丹、炼器、制符、阵法）
- [ ] 设施升级（洞府等级、灵田产出、炼丹炉品质）

#### 3.3 战斗系统
- [ ] 回合制战斗（移植现有战斗计算逻辑）
- [ ] 潜行机制（《这是我的战争》风格：避免正面冲突）
- [ ] Buff/Debuff 系统

#### 3.4 心理/心境系统
- [ ] 心境值（类似 TWoM 的心情系统）
- [ ] 道德抉择影响（正邪选择 → 心境波动 → 修炼效率变化）
- [ ] 心魔系统（心境过低触发心魔，影响突破成功率）

### 阶段四：子系统移植（8-10 周）

**目标**：按优先级逐个移植子系统

| 优先级 | 子系统 | 说明 |
|--------|--------|------|
| P0 | 角色/NPC 系统 | 好感度、关系网、记忆 |
| P0 | 世界演化 | AI 驱动事件生成 |
| P1 | 探索引擎 | 地图探索、遭遇计算 |
| P1 | 门派系统 | 宗门建设、弟子管理 |
| P2 | 图片生成 | NPC/场景文生图 |
| P2 | Galgame 模式 | 分支叙事、CG 画廊 |
| P3 | 存读档+云同步 | 本地存档 + GitHub 同步 |
| 暂缓 | NSFW 子系统 | 大量现代场景，不适合 Godot 版本首期 |
| 暂缓 | 桌游系统 | 大富翁/狼人杀等，独立性强 |
| 暂缓 | 移动设备模拟 | 20+ 虚拟 App，工作量巨大 |

### 阶段五：打磨与优化（4-6 周）

- [ ] 音效与音乐（氛围音效、BGM）
- [ ] 粒子特效（修炼、战斗、炼丹）
- [ ] 动画系统（角色行走、交互、战斗动画）
- [ ] 性能优化（场景加载、资源管理）
- [ ] 多平台导出（PC、Android、Web）

## 五、技术架构建议

### 5.1 Godot 项目结构

```
morankun.godot/
├── scenes/
│   ├── main/              # 主场景
│   │   ├── game_main.tscn       # 游戏主场景（2D 横截面视图）
│   │   ├── shelter.tscn         # 居所场景
│   │   └── exploration.tscn     # 探索场景
│   ├── ui/                # UI 场景
│   │   ├── hud.tscn             # 主 HUD
│   │   ├── inventory.tscn       # 背包界面
│   │   ├── character.tscn       # 角色面板
│   │   ├── dialogue.tscn        # 对话/叙事面板
│   │   └── settings.tscn        # 设置界面
│   └── characters/        # 角色场景
│       ├── player.tscn          # 玩家角色
│       └── npc.tscn             # NPC
├── scripts/
│   ├── autoload/          # 全局单例
│   │   ├── game_manager.gd
│   │   ├── ai_manager.gd
│   │   ├── save_manager.gd
│   │   └── event_bus.gd
│   ├── core/              # 核心逻辑
│   │   ├── character.gd
│   │   ├── item.gd
│   │   ├── world_state.gd
│   │   └── day_night_cycle.gd
│   ├── systems/           # 子系统
│   │   ├── battle_system.gd
│   │   ├── crafting_system.gd
│   │   ├── morality_system.gd
│   │   ├── exploration_system.gd
│   │   └── shelter_system.gd
│   └── ui/                # UI 脚本
├── resources/             # 游戏资源
│   ├── themes/            # Godot Theme
│   ├── fonts/             # 字体
│   └── shaders/           # 着色器
├── data/                  # 数据文件
│   ├── presets/           # 天赋、背景、修炼体系预设 (JSON)
│   ├── prompts/           # AI 提示词模板 (JSON)
│   └── eras/              # 纪元配置 (JSON)
└── assets/                # 美术资源
    ├── sprites/
    ├── tiles/
    ├── ui/
    ├── audio/
    └── animations/
```

### 5.2 关键技术选型

| 需求 | 方案 | 说明 |
|------|------|------|
| 2D 渲染 | Godot 内置 2D | TileMap + Sprite2D + CanvasLayer |
| AI 调用 | HTTPRequest + JSON | 支持流式响应（SSE） |
| 数据持久化 | FileAccess + JSON | 本地存档，可选云同步 |
| 事件通信 | Godot Signal 系统 | 替代 React 的事件总线 |
| 状态管理 | Autoload 单例 | 全局状态管理 |
| 动画 | AnimationPlayer + SpriteFrames | 2D 帧动画 |
| 特效 | GPUParticles2D | 粒子特效 |

## 六、风险评估

| 风险 | 等级 | 应对措施 |
|------|------|---------|
| AI 调用延迟影响体验 | **高** | 本地缓存常用响应、异步加载、加载动画 |
| 2D 美术资源需求量大 | **高** | 先用 AI 生成 placeholder，后期替换 |
| 项目范围过大 | **高** | 严格按阶段推进，首期只保留核心玩法 |
| Godot 4 GDScript 学习曲线 | **中** | TypeScript → GDScript 迁移成本中等 |
| 流式 AI 响应解析 | **中** | 复用现有 JSON 修复逻辑 |
| 多纪元主题差异化 | **中** | 每个纪元独立 TileSet 和 Theme |

## 九、核心移植映射表

以下是现有 Web 项目核心模块到 Godot 的映射关系：

| Web 项目模块 | Godot 对应实现 | 迁移方式 |
|-------------|---------------|---------|
| `hooks/useGame.ts` | `autoload/game_manager.gd` | 核心逻辑转写 |
| `hooks/useGame/sendWorkflow.ts` | `systems/narrative_system.gd` | AI 叙事循环 |
| `hooks/useGame/systemPromptBuilder.ts` | `systems/prompt_assembler.gd` | 提示词组装 |
| `services/ai/` | `autoload/ai_manager.gd` | HTTP 请求重写 |
| `services/dbService.ts` | `autoload/save_manager.gd` | FileAccess 替代 |
| `models/domain/角色.ts` | `core/character.gd` (Resource) | 类型转写 |
| `models/domain/物品.ts` | `core/item.gd` (Resource) | 类型转写 |
| `components/features/Chat/` | `scenes/ui/dialogue.tscn` | UI 重建 |
| `components/features/Inventory/` | `scenes/ui/inventory.tscn` | UI 重建 |
| `prompts/runtime/` | `data/prompts/` (JSON) | 直接复用 |
| `data/presets.ts` | `data/presets/` (JSON) | 直接复用 |
| `models/eraTheme/` | `data/eras/` (JSON) + Theme | 配置+主题 |
| Zustand 状态管理 | Autoload 单例 + Signal | 架构替换 |
| React 组件树 | Godot Scene 树 | 完全重建 |
